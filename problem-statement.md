# Problem Statement — URL Shortener Service

---

## Section 1 — The Problem

Long URLs are unwieldy in messages, printed materials, and social media posts. A URL
shortening service converts any long URL into a short, opaque code that redirects to the
original destination. The value proposition is simple: short links are shareable, trackable,
and can be branded with custom aliases. The engineering challenge is that what appears to be
a trivially simple service — store a URL, return a short code, redirect on lookup — becomes
genuinely hard at the scale of hundreds of millions of redirects per day, where the read
path must be as fast as physically possible and the write path must generate collision-free
short codes at high throughput.

---

## Section 2 — Why It Is Hard

- **Read-to-write asymmetry**: URL shorteners are read-dominated by a ratio of roughly
  100:1 or higher. A URL created once may be redirected millions of times. The architecture
  must be optimised almost entirely for the read path — the redirect must resolve in
  single-digit milliseconds, which means the database cannot be on the critical path for
  the vast majority of requests.

- **Short code uniqueness at scale**: Generating a random short code and checking for
  collisions in the database works until it does not — at hundreds of millions of URLs,
  collision probability becomes non-trivial. The generation strategy must either guarantee
  uniqueness without a database roundtrip, or use a pre-generated pool that eliminates
  real-time collision resolution from the hot path.

- **Redirect latency requirements**: Users following a short link expect to land on the
  destination page without perceiving any intermediate step. Sub-10-millisecond redirect
  resolution requires the short code to URL mapping to be served from memory — a Redis
  cache in front of the database, or ideally a CDN edge cache for the hottest links.

- **Click analytics at scale**: Recording a click event for every redirect at 100M
  redirects/day is 1,157 writes per second to the analytics pipeline. This cannot be
  synchronous on the redirect path — it must be fire-and-forget, queued for async
  processing, or batched at the edge.

- **Custom alias conflicts**: When users specify a custom alias, the system must check
  for conflicts against both existing short codes and reserved words (system paths, API
  routes, offensive terms) atomically, without creating a race condition under concurrent
  alias creation.

---

## Section 3 — Scope of This Implementation

**In scope:**
- Short code generation using base62 encoding (8 characters, ~218 trillion combinations)
- Collision-safe code generation with database uniqueness enforcement
- Custom alias support with reserved word validation
- Redirect service with Redis cache-aside (TTL: 10 minutes for standard links)
- URL expiry: optional `expires_at` field, expired URLs return 410 Gone
- Click event recording via async queue (fire-and-forget on redirect path)
- Basic click analytics: total clicks, clicks by day, referrer breakdown
- URL deactivation (soft delete — short code returns 404 when deactivated)

**Out of scope:**
- CDN edge caching integration
- QR code generation
- Link preview / Open Graph metadata proxying
- A/B testing with multiple destination URLs per short code
- Branded domain support (custom domains per user/organisation)
- Real-time analytics dashboard

---

## Section 4 — Success Criteria

The system is working correctly when:

1. Every `GET /{code}` request for a valid, active, non-expired short code returns a 301
   redirect to the correct destination URL within 10ms at p99 under production load.

2. Two concurrent requests to create URLs with the same custom alias result in exactly one
   success and one 409 Conflict — never two records with the same alias.

3. A short code generated by the system is globally unique — no two URLs share the same
   short code at any point in time.

4. Click events are recorded for analytics without any impact on redirect response time —
   the redirect path never waits for the analytics write to complete.

5. An expired URL (past its `expires_at`) returns 410 Gone, not a redirect, and does so
   without a cache serving stale data beyond a single TTL window.

6. The system sustains 100 million redirects per day (approximately 1,200 requests per
   second sustained, with burst peaks) with cache hit rates above 90% for active links.
